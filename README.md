## Запуск локально
```bash
make up
make migrate
make createsuperuser
```
Swagger: http://localhost:8000/api/docs

Админка: http://localhost:8000/admin/


Для базового функционала управления пользователями текущих эндпоинтов достаточно, однако для полноценной работы платформы разметки данных, описанной в архитектуре, вам не хватает слоев управления **задачами**, **пуллами** и **администрирования**[cite: 1].

Ниже приведен план развития API с учетом вашей модели `Pool` и требований к динамическому назначению задач.

---

### 1. Анализ текущих эндпоинтов
У вас реализован базовый CRUD для профиля. Для «института» и «платформы разметки» стоит добавить:
*   **DELETE `/api/users/me`** — не удаление, а скорее деактивация аккаунта студентом.
*   **GET `/api/users/{uuid}`** — просмотр профиля другого пользователя (нужен для администраторов и валидаторов).

---

### 2. Новые эндпоинты для модели Pool (Пулы задач)
Пул — это группировка задач, объединенных общими требованиями к навыкам (`skills`)[cite: 1].
*   **GET `/api/pools/`** — список всех пулов (фильтрация по `is_active`).
*   **POST `/api/pools/`** — создание нового пула (только для роли `ADMIN`).
*   **GET `/api/pools/{pool_id}`** — детальная информация о пуле (статистика, требуемые скиллы).
*   **PATCH `/api/pools/{pool_id}`** — закрытие пула или изменение `overlap` (количества людей, которые должны разметить одну задачу).

---

### 3. Эндпоинты для разметки (Кабинет Студента)
Согласно требованиям, студент должен получать задачи динамически (одну за раз)[cite: 1].
*   **POST `/api/assignments/next`** — самый важный эндпоинт. Система должна найти подходящий активный `Pool` (по скиллам студента), выбрать свободную задачу и создать запись в таблице `assignments`[cite: 1].
*   **PATCH `/api/assignments/{assignment_id}`** — отправка результата разметки (сохранение JSONB данных аннотации)[cite: 1].
*   **GET `/api/assignments/my`** — история выполненных задач студентом и его текущий прогресс.

---

### 4. Эндпоинты для Валидаторов и Админов
Поскольку в вашей системе есть роль `VALIDATOR`[cite: 1]:
*   **GET `/api/assignments/pending`** — список выполненных задач, ожидающих проверки.
*   **POST `/api/assignments/{assignment_id}/verify`** — вынесение вердикта валидатором (принять/отклонить).
*   **GET `/api/stats/`** — общая статистика по датасетам и прогрессу студентов для админ-панели[cite: 1].

---

### Приоритетность разработки (Что делать следующим?)

Рекомендую двигаться по логике **"Путь задачи"**:

1.  **Управление пуллами (`/api/pools/`)**: Чтобы было откуда брать задачи.
2.  **Динамическая выдача (`/api/assignments/next`)**: Реализация логики `Task Manager`, которая сопоставляет `Skill` пользователя и `Skill` пула[cite: 1].
3.  **Сохранение результата (`PATCH /api/assignments/{id}`)**: Работа с полем `JSONB` для хранения координат полигонов и масок[cite: 1].

### Архитектурный совет по Pool
В модели `Pool` у вас есть поле `overlap`[cite: 1]. При реализации эндпоинта выдачи задачи (`next`) вам нужно будет считать количество уже созданных `assignments` для конкретной задачи, чтобы оно не превышало `overlap` из соответствующего пула.

Какие эндпоинты нужны для задач?

Учитывая требования к «динамическому назначению» и наличие «кабинета администратора», вот необходимый набор:
1. Для Администратора (Управление контентом)

    POST /api/tasks/ — загрузка новых задач (обычно массовая через ZIP или ссылки на MinIO).

    GET /api/tasks/ — список всех задач с фильтрацией по pool_id или dataset_id.

    DELETE /api/tasks/{task_id} — удаление бракованной задачи.

2. Для системы (Логика выдачи)

Вам не нужен эндпоинт GET /api/tasks/{id} для студента, так как он не должен сам выбирать задачу по ID. Вместо этого используется эндпоинт назначения:

    POST /api/assignments/next — внутренняя логика этого метода обращается к модели Task, находит ту, у которой количество текущих ассигнований меньше, чем overlap в её пуле, и отдает её студенту.

3. Для Студента (Просмотр текущей работы)

    GET /api/assignments/current — возвращает детали задачи (включая image_url), которую студент размечает в данный момент.
